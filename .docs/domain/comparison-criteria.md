# 비교 기준 정의

## 문서 목적

이 문서는 컨텍스트 엔지니어링과 하네스 엔지니어링을 비교하기 위한 기준을 정의한다.

이 프로젝트의 비교는 단순한 인상 평가가 아니라, 동일한 시나리오에서 어떤 방식이 더 재현 가능하고, 검증 가능하며, 설명 가능한 결과를 만드는지 구조적으로 판단하기 위한 것이다.

## 기준 사용 원칙

- 모든 비교는 동일한 기준 집합을 사용한다
- 점수만 기록하지 않고 근거를 함께 남긴다
- 비교 기준은 설명 가능성을 우선한다
- 초기 버전에서는 기준 수를 제한해 해석 가능성을 유지한다

## 비교 기준

### Reproducibility

동일한 시나리오와 유사한 조건에서 비슷한 결과를 다시 만들어낼 수 있는 정도다.

중요한 이유:

- 같은 에이전트를 사용해도 결과가 흔들리는 문제를 확인하려는 프로젝트 목적과 직접 연결된다
- 결과 차이를 우연이 아니라 구조 차이로 해석하기 위해 필요하다

기록 방식:

- 반복 실행 가능 여부
- 실행 조건 고정 여부
- 결과 편차 메모

### Verifiability

결과가 좋은지 나쁜지를 기준에 따라 판단할 수 있는 정도다.

중요한 이유:

- 결과 생성 자체보다 결과 검증 가능성이 더 중요할 수 있다
- 하네스 엔지니어링의 강점이 드러날 가능성이 큰 기준이다

기록 방식:

- 검증 규칙 존재 여부
- 자동 검증 가능 여부
- 수동 평가 메모
- 검증 통과 여부

### Maintainability

구조나 요구사항이 바뀌었을 때 수정 범위와 변경 비용을 관리하기 쉬운 정도다.

중요한 이유:

- 실무에서는 단발성 결과보다 이후 수정 가능성이 중요하다
- 제약과 재시도 구조를 가진 방식이 유지보수에 어떤 영향을 주는지 확인할 수 있다

기록 방식:

- 수정 대상 범위
- 설정 분리 수준
- 구조 변경 메모

### Scalability

새로운 시나리오, 비교 항목, 실행 조건이 추가될 때 구조적으로 확장하기 쉬운 정도다.

중요한 이유:

- 초기 실험은 작아도 프로젝트가 확장될수록 구조 차이가 더 크게 드러난다
- 단순 컨텍스트 추가 방식과 구조적 하네스 방식의 차이를 비교하기 좋다

기록 방식:

- 시나리오 추가 난이도
- 기준 추가 난이도
- 구조 확장 메모

### Explainability

왜 해당 결과가 나왔는지 근거와 흐름을 설명하기 쉬운 정도다.

중요한 이유:

- 이 프로젝트는 단순 성공률보다 결과를 설명 가능한 구조로 남기는 것을 중요하게 본다
- 포트폴리오와 면접에서 설계 의도를 설명하는 데 직접 연결된다

기록 방식:

- 근거 데이터 존재 여부
- 결과 해석 메모
- 실패 원인 기록 여부

## 기록 데이터 방향

초기 버전에서는 아래 두 종류의 데이터를 함께 기록한다.

### 정성 데이터

- 평가 메모
- 실패 원인
- 구조적 해석
- 비교 코멘트

### 정량 또는 반정량 데이터

- 성공 여부
- 검증 통과 여부
- 수정 횟수
- 재시도 횟수
- 단계 수
- 점수 또는 등급

## 문서 역할

이 문서는 이후 `Evaluation`, `Metric`, `Evidence` 도메인을 정의할 때 기준 문서로 사용한다.
