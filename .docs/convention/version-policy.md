# 버전 관리 정책

## 목적

이 문서는 `harness-lab` 프로젝트의 버전관리 기준을 정의한다.

이 프로젝트는 구현 결과뿐 아니라 설계 과정 자체도 포트폴리오의 일부이므로, 문서와 코드의 변경 이력이 모두 의미를 가진다.

버전 정책은 사람이 일관되게 판단하기 위한 기준이면서, 이후 `AGENT.md`에서 에이전트가 변경 영향도를 해석할 때 참고하는 기준이기도 하다.

## 기본 원칙

- 작은 단위의 커밋을 우선한다
- 하나의 커밋은 하나의 의도를 가져야 한다
- 문서 변경도 코드 변경만큼 중요하게 관리한다
- 구현 전에 설계 문서를 먼저 정리하고, 이후 구현 변경과 연결한다

## 브랜치 전략

이 프로젝트는 개인 프로젝트이므로 단순한 브랜치 전략을 사용한다.

- `main`: 정리된 기준 상태 유지
- `docs/*`: 문서 작업
- `feature/*`: 기능 구현 작업

예시:

- `docs/readme`
- `docs/domain-overview`
- `feature/scenario-entity`

## 커밋 메시지 규칙

아래 형식을 기본으로 사용한다.

```text
<type>: <summary>
```

권장 type:

- `docs`
- `feat`
- `fix`
- `refactor`
- `chore`
- `test`

예시:

- `docs: README 초안 작성`
- `docs: 프로젝트 개요 문서 추가`
- `feat: scenario 도메인 추가`
- `fix: experiment 상태 값 검증 수정`

## 문서 관리 원칙

- README는 프로젝트의 입구 역할만 수행한다
- 상세 설계는 `.docs/` 하위에 역할별로 분리한다
- 문서 제목은 기능보다 의도를 드러내야 한다
- 문서가 바뀌면 커밋 메시지에도 그 변경 이유가 드러나야 한다

## 관리 기준

- 문서 작업과 구현 작업은 가능한 한 분리된 커밋으로 관리한다
- 비교 기준이나 도메인 정의가 바뀌면 관련 문서를 먼저 수정한 뒤 코드 변경을 반영한다
- README와 `.docs` 사이에 중복이 생기면 README는 요약, `.docs`는 상세 설명을 맡는다

## 버전 판정 기준

이 프로젝트는 Semantic Versioning의 개념을 참고하되, 개인 프로젝트 특성에 맞게 다음 기준으로 해석한다.

## 0.x.x 운영 기준

이 프로젝트는 초기 단계에서 `0.x.x` 버전을 사용한다.

`1.0.0` 이전 버전은 아직 구조와 기능이 고정되지 않은 상태로 보고, 아래 기준으로 해석한다.

### 0.0.x

아래와 같은 변경에 사용한다.

- 오타 수정
- 문서 표현 보완
- 설정 수정
- 구조에 큰 영향이 없는 소규모 정리

즉, 프로젝트의 방향은 유지되지만 완성도만 조금씩 높아지는 변경에 해당한다.

### 0.1.0

아래와 같은 첫 의미 있는 설계 기준 확정에 사용한다.

- README와 `.docs` 구조가 정리된 경우
- 도메인 정의와 비교 기준이 프로젝트 기준으로 고정되기 시작한 경우
- 이후 구현의 기준이 되는 설계 문서 집합이 갖춰진 경우

즉, 프로젝트의 뼈대가 처음으로 설명 가능한 형태를 갖춘 시점을 의미한다.

### 0.2.0 이상

아래와 같이 기능 또는 구조가 단계적으로 확장될 때 사용한다.

- 핵심 도메인 엔티티 구조 확정
- API 초안 또는 실제 API 추가
- 비교 결과 저장/조회 기능 추가
- 화면 또는 배포 구조 추가

즉, 프로젝트가 설계 문서 중심 단계를 넘어 실제 동작 가능한 구조로 확장될 때 minor 증가를 사용한다.

### 1.0.0

아래 조건을 만족할 때 첫 안정 버전으로 본다.

- 핵심 도메인 구조가 확정되어 있고
- 주요 기능이 실제로 동작하며
- README, `.docs`, `CHANGELOG`가 공개 저장소 기준으로 정리되어 있고
- 포트폴리오 제출 또는 공개 시연이 가능한 상태인 경우

즉, 더 이상 단순 초안이 아니라 하나의 완성된 개인 프로젝트로 설명할 수 있는 시점을 의미한다.

### Major

아래와 같은 경우 `major` 후보로 본다.

- 기존 API 계약이 깨지는 변경
- 핵심 도메인 구조가 이전 방식과 호환되지 않게 바뀌는 변경
- 기존 사용 흐름이나 데이터 해석 방식이 근본적으로 달라지는 변경

### Minor

아래와 같은 경우 `minor` 후보로 본다.

- 기존 동작을 유지하면서 기능이나 문서 구조가 의미 있게 확장되는 변경
- 새로운 API, 화면, 도메인 기능이 추가되는 변경
- 프로젝트 사용 방식은 유지되지만 비교 구조나 조회 기능이 확장되는 변경

### Patch

아래와 같은 경우 `patch` 후보로 본다.

- 버그 수정
- 오타 수정
- 문장 표현 개선
- 설정 보완
- 동작을 깨지 않는 소규모 리팩터링

### No Version Change

아래와 같은 경우 버전 증가 없이 changelog만 기록하거나, changelog도 생략할 수 있다.

- 내부 메모 수준의 문서 수정
- 설명 보강만 있고 프로젝트 사용 방식에 영향이 없는 변경
- 실험 중간 정리처럼 외부에 의미 있는 변화가 아닌 경우

## Changelog 반영 원칙

- 사용자 관점에서 의미 있는 변화가 있으면 `CHANGELOG.md`에 기록한다
- 버전 영향이 불명확하면 `Version Impact`를 `pending`으로 둔다
- 에이전트는 변경 후 `major`, `minor`, `patch`, `no change` 중 하나를 제안할 수 있지만, 최종 버전 확정은 사용자의 판단을 따른다
- `0.x.x` 구간에서는 현재 프로젝트 성숙도를 함께 고려해 `0.0.x`, `0.1.0`, `0.2.0` 이상의 형태로 제안할 수 있다
